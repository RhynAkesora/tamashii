# プロジェクトルール

## Git操作
- push、pull、commit、merge などの git 操作は、できる限りアシスタント側で自主的に実行すること
- ユーザーに git 操作を委ねず、作業完了時に自動的に commit & push まで行う

## バージョン管理
- index.html を1行でも変更したら、バージョンを 0.001 上げること（確認不要）
- バージョン表記は index.html 内の `ctx.fillText('Ver. X.XXX', ...)` を更新する
- バージョン表記はタイトル画面と設定画面の2箇所に存在するため、更新時は必ず両方を同時に更新すること
- service-worker.js の `CACHE_NAME` もバージョンに合わせて更新すること（例: `tamashii-no-kyoumei-v3724`）
- 応答の末尾に現在のバージョンを表記すること（例: `現在のバージョン: 3.720`）

## リリース準備作業（段階的対応）
- 正式リリースに向けた全タスクは `RELEASE_CHECKLIST.md` に記載されている
- コンテキスト制限があるため、1セッションで全タスクを完了する必要はない
- 各セッションではフェーズ番号順に、1〜2フェーズ分のタスクを対応する
- タスク完了時は `RELEASE_CHECKLIST.md` のチェックボックスを `[x]` に更新する
- セッション開始時に `RELEASE_CHECKLIST.md` を読み、未完了タスクの続きから着手する
- 作業前に必ず対象コードの現在の行番号を確認すること（行番号は変更により変動する）
- フェーズ1（クリティカルバグ）→ フェーズ2（中優先度バグ）→ … の順で進める

## ストアリリースガイドライン（iOS App Store / Google Play Store）

本アプリは Capacitor を用いた PWA ハイブリッドアプリとして、両ストアへのリリースを目指す。
以下の基準を満たすよう、開発・修正時に常に意識すること。

### 共通要件（iOS / Android 両方）

#### アプリの品質・安定性
- クラッシュ・フリーズ・ANR（Application Not Responding）がないこと
- 全画面サイズ・解像度で正しく表示されること（レスポンシブ対応）
- オフライン時に適切なエラーハンドリングがあること（Service Worker活用）
- メモリリークがないこと（特に Canvas / GPU リソースの解放）
- バックグラウンド復帰時に正常動作すること

#### プライバシー・データ収集
- プライバシーポリシーを用意し、ストア掲載情報およびアプリ内からアクセス可能にすること
- 収集するデータの種類（Firebase Analytics、AdMob広告ID等）を正確に申告すること
- ユーザーの個人データを収集する場合は事前に同意を取得すること
- GDPR / CCPA 等の地域別プライバシー規制に準拠すること

#### 広告（AdMob）
- 広告がゲームプレイを過度に妨げないこと（インタースティシャルの頻度に注意）
- 広告の表示位置がインタラクティブ要素と重ならないこと（誤タップ防止）
- 子供向けコンテンツに該当する場合は COPPA / 子供向け広告設定を有効にすること
- テスト広告IDを本番ビルドに含めないこと

#### コンテンツガイドライン
- 著作権侵害コンテンツを含まないこと（画像・音声・フォント等のライセンス確認）
- 不適切なコンテンツ（暴力・性的・差別的表現）を含まないこと
- 年齢レーティングを正確に設定すること（本アプリはパズルゲームのため全年齢向け想定）

#### メタデータ・ストア掲載情報
- アプリ名・説明文・スクリーンショットがアプリの実際の機能を正確に反映すること
- スクリーンショットは実機の画面キャプチャを使用すること（モックアップ不可）
- キーワードスパム（無関係なキーワードの羅列）をしないこと

### iOS 固有要件（Apple App Store Review Guidelines）

#### 技術要件
- `PrivacyInfo.xcprivacy` に Required Reason API の使用目的を正確に記載すること
- App Transport Security (ATS) に準拠すること（HTTPS必須）
- IPv6 ネットワークで動作すること
- 最新の Xcode / iOS SDK でビルドすること
- UIWebView は使用不可（WKWebView を使用すること — Capacitor 6+ はデフォルトで対応済み）

#### デザイン・UX
- Apple Human Interface Guidelines に著しく反しないこと
- ステータスバー・ノッチ・Dynamic Island を適切に処理すること（Safe Area対応）
- 最低限のネイティブ機能（WebView だけのラッパーアプリは審査でリジェクトされるリスクあり）
  - オフライン機能、プッシュ通知、ハプティクス等でネイティブ価値を示すこと
- iPadでも適切に動作すること（Universal App の場合）

#### 課金・広告
- アプリ内課金がある場合は Apple IAP を使用すること（外部決済リンク不可）
- 「復元」ボタンの実装（課金がある場合）
- SKAdNetwork 対応（広告アトリビューション）

#### プライバシー
- App Tracking Transparency (ATT) フレームワークの実装（IDFA使用時は必須）
- App Store Connect のプライバシーラベル（Data Safety）を正確に記入すること

### Android 固有要件（Google Play Store ポリシー）

#### 技術要件
- `targetSdkVersion` を Google Play が要求する最新レベルに合わせること
- 64-bit アーキテクチャ対応（AAB形式で提出）
- Android App Bundle (AAB) 形式での提出が必須
- ProGuard / R8 による難読化・最適化の適用を推奨

#### デザイン・UX
- Material Design ガイドラインに著しく反しないこと
- 戻るボタン（バックキー）の適切な処理
- 画面回転・マルチウィンドウ時の動作確認
- エッジ・ジェスチャーナビゲーション対応

#### 広告・課金
- Android 用の AdMob ユニットIDを設定すること（iOS用IDのフォールバックは不可）
- Google Play 課金ライブラリの使用（課金がある場合）

#### プライバシー・安全性
- Data Safety セクションの正確な記入
- パーミッションの最小化（不要な権限を要求しないこと）
- ターゲットオーディエンスの設定（子供向けの場合はファミリーポリシー準拠）

### ストアリリース前の最終チェック項目
- [ ] プライバシーポリシーページの作成・公開
- [ ] 全対象デバイスでの動作確認（iPhone SE〜Pro Max、各種Android端末）
- [ ] 本番用 AdMob ユニットID（iOS / Android 各1つ）の設定
- [ ] スクリーンショット撮影（iOS: 6.7" & 5.5"、Android: 各種サイズ）
- [ ] アプリアイコン全サイズの準備（1024x1024 マスターアイコン含む）
- [ ] 年齢レーティング質問票の回答準備
- [ ] ATT（App Tracking Transparency）ダイアログの実装確認
- [ ] Firestore Security Rules のサーバー側バリデーション確認
- [ ] テスト広告ID → 本番広告ID への切り替え確認
- [ ] リリースビルドでの最終動作確認（Debug フラグ / console.log の除去）
